## КРИТИЧЕСКИЕ БЛОКЕРЫ (нужно исправить до начала разработки)

### КР-01 · Wall не имеет обратных ссылок для отношений Angle

**Проблема**: В `technical-requirements.md` модель `Angle` ссылается на `Wall` дважды через именованные отношения:
```prisma
wallAId String
wallBId String
wallA   Wall   @relation("angleWallA", fields: [wallAId], references: [id])
wallB   Wall   @relation("angleWallB", fields: [wallBId], references: [id])
```
Prisma **обязательно** требует обратных ссылок в модели `Wall`:
```prisma
anglesAsWallA Angle[] @relation("angleWallA")
anglesAsWallB Angle[] @relation("angleWallB")
```
Без них `prisma generate` / `prisma migrate` упадёт с ошибкой компиляции схемы.

**Файлы**: `docs/technical-requirements.md` (секция 4.9), `sprints/sprint-1.md` (задача S1-03)  
**Исправление**: Добавить две строки в модель `Wall` в `technical-requirements.md`.

---

### КР-02 · seed: adjacency с `hasDoorBetween: true` не содержит `doorOpeningId`

**Проблема**: В `seed/test-apartment.json` все 4 смежности с `hasDoorBetween: true` имеют `doorOpeningId: null`:
```json
{ "wallALabel": "A", "wallBLabel": "A", "hasDoorBetween": true, "doorOpeningId": null }
```
В `technical-requirements.md` явно задан инвариант:
> Если `hasDoorBetween = true`, `doorOpeningId` должен быть не null.

Кроме этого, `onDelete: SetNull` на `doorOpeningId` означает, что при удалении двери `hasDoorBetween` останется `true` с `null`-ссылкой — инвариант нарушается и при runtime-удалении.

**Файлы**: `seed/test-apartment.json`, `docs/technical-requirements.md` (секция 4.8)  
**Исправление (2 шага)**:
1. В seed использовать `_doorOpeningIndex` (аналогично `_windowOpeningIndex`) для двухфазного заполнения.
2. В схеме добавить триггер или хук (Prisma middleware) для обнуления `hasDoorBetween` при SetNull.

### КР-03 · DoorOpening не содержит полей откосов (reveals)

**Проблема**: В `use-cases.md` шаг UC-03.5 явно описывает:
> Для каждой двери: откосы (глубина ниши слева/справа)

Модель `DoorOpening` в `technical-requirements.md`:
```prisma
model DoorOpening {
  id              String  // ...
  width           Int
  heightFromScreed Int
  photoPath       String?
  // ... НЕТ полей откосов
}
```
Данные будут потеряны — некуда записывать.

**Файлы**: `docs/technical-requirements.md` (секция 4.7), `docs/use-cases.md` (UC-03.5)  
**Исправление**: Добавить в модель `DoorOpening`:
```prisma
revealLeft  Int?  // мм, глубина откоса слева
revealRight Int?  // мм, глубина откоса справа
```
И соответствующий API-контракт в секции 6 tech-req.

### КР-04 · WallAdjacency не поддерживает частичное перекрытие стен

**Проблема**: В seed квартира содержит смежность Коридор↔Кухня, где стены имеют разную длину (725мм vs 3390мм). В `technical-requirements.md` секция 8 (PlanAssembler) упрощение зафиксировано:
> стыкуются по начальной точке

Но в модели `WallAdjacency` нет полей для хранения смещения:
```prisma
model WallAdjacency {
  // ... нет offsetInLongerWall, alignmentType и т.д.
}
```
Если в реальных данных пользователя понадобится выравнивание не по началу — данные не сохранить, план будет неверным.

**Оценка риска**: СРЕДНИЙ для MVP (упрощение задокументировано), но ВЫСОКИЙ для расширения.  
**Рекомендация**: Задокументировать ограничение в business-requirements и добавить `alignmentType: ALIGN_START | ALIGN_CENTER | ALIGN_CUSTOM` и `wallBOffset Int?` с дефолтом `ALIGN_START` для будущей совместимости.

### В-02 · PIPE-элемент в seed использует `wallLabel`, но схема не требует `wallId` для труб

В seed:
```json
{ "type": "PIPE", "wallLabel": "A-B", "positionX": 300, ... }
```
В таблице `RoomElement` (tech-req секция 4.10) для типа `PIPE` поле `wallId` помечено как `—` (необязательно). Труба при этом должна привязываться к стене через `wallLabel` в seed, но в реальной схеме нет валидации этого отношения.

**Риск**: Seed-скрипт будет работать, но при создании через API трубы могут создаваться без привязки к стене, а UI не сможет правильно отображать их позицию.

**Рекомендация**: Уточнить в tech-req — является ли `wallId` обязательным для `PIPE`. Если да — добавить CHECK в схему или валидацию в NestJS.

---

### В-03 · COLUMN не имеет координат в пространстве комнаты

В seed элемент `COLUMN` имеет только `positionX` (относительно стены). Для отдельно стоящей колонны (посреди комнаты) нужны координаты `(x, y)` в системе комнаты.

В tech-req секция 4.10 `positionX / positionY` описаны как:
> positionX — смещение от начала стены (мм)

Для свободно стоящего COLUMN этого недостаточно.

**Рекомендация**: Добавить `roomPositionX Int?`, `roomPositionY Int?` для элементов без привязки к стене.

Enum `PhotoType` содержит `OVERVIEW_BEFORE` и `OVERVIEW_AFTER`. Нигде в документах не определено: "до/после чего"? Ремонта? Съёмки? Это создаёт неоднозначность при реализации и UI.

**Рекомендация**: Переименовать или документировать семантику в tech-req секции 4.11.

- Площадь: 3.791 × 2.450 = **9.288 м²** (seed: 9.29 ✓)

- Объём (h=2700): 5.182 × 2.700 = **13.992 м³** (seed: 13.99 ✓)

### Коридор L-образный (CORRIDOR)
- Прямоугольник: 1361мм × (2310+351)мм минус вырез (1361-1010)мм × 351мм
- = 1.361 × 2.661 − 0.351 × 0.351 = 3.621 − 0.123... 

  *Пересчёт по координатам стен*:
  - A: 1361, B: 725, C: 1010, D: 725 (до угла D), E: 351, F: 2310
  - Формула Гаусса для 6-угольника: |½Σ(xᵢyᵢ₊₁ − xᵢ₊₁yᵢ)|
  - Упрощённо: основной прямоугольник 1361×2310 = 3,144,510 минус вырез 351×725 = 254,475
  - **Площадь = (3,144,510 − 254,475) / 1,000,000 = 2.890 м²** (seed: 2.89 ✓)
- Периметр: (1361+725+1010+725+351+2310)/1000 = **6.482 м** (seed: 6.48 ✓)

## АРХИТЕКТУРНЫЕ НАБЛЮДЕНИЯ

### Хорошо спроектировано ✅
- Единица хранения — миллиметры (Int в DB) с конвертацией только на UI-слое: защита от float-ошибок
- UUID as String для SQLite: обоснованный trade-off (нет BINARY(16) в SQLite)
- Двухфазный seed с `_index`-ссылками: правильное решение для circular FK
- Каскадные удаления везде: Project → Room → Wall → всё. Чистая модель.
- Adjacency граф + Shoelace: математически корректный подход для 2D-плана

### Требует внимания ⚠️
- `FloorPlanLayout` хранит `layoutJson: String` — непрозрачный JSON-блоб без схемы. При изменении формата данных нет пути миграции без ручной правки всех записей.  
  **Рекомендация**: Добавить поле `layoutVersion Int @default(1)` для управления миграциями формата.
- Нет soft-delete. При удалении проекта все фото физически удаляются (через FileStorageService). Если есть ошибка в API — данные невозможно восстановить.  
  **Рекомендация**: Для MVP достаточно документировать это явно в business-req как "удаление необратимо".
- `WallSegment` без `order` поля — сортировка по `createdAt` ненадёжна при быстрых операциях или миграциях данных.  
  **Рекомендация**: Добавить `order Int` в `WallSegment`.
