## Критические проблемы (блокеры)

### КР-01 — Seed: сегменты DOOR/WINDOW без FK на Opening

**Файл**: `seed/test-apartment.json`  
**Серьёзность**: БЛОКЕР (Integration Test)

Сегменты с `type: "DOOR"` и `type: "WINDOW"` не имеют полей `doorOpeningId` / `windowOpeningId`. По `technical-requirements.md` валидация требует, чтобы эти FK были заполнены.

**Решение**: использовать двухфазный seed — сначала создать Wall + WallSegment (type=SOLID), затем создать Opening и обновить соответствующие сегменты, добавив FK.

---

### КР-02 — Seed: нет объектов Angle для L-образного коридора

**Файл**: `seed/test-apartment.json`  
**Серьёзность**: БЛОКЕР (Plan Assembler)

Алгоритм `PlanAssemblerService` строит полигоны через угловые объекты. Коридор — L-образный (6 вершин). В seed нет ни одного объекта `Angle`. Без них алгоритм не может различить выпуклые (90°) и вогнутые (270°) углы — план будет неверным.

**Решение**: добавить в seed объекты `Angle` для каждой вершины коридора, явно указывая `angle: 90` или `angle: 270`.

---

### КР-03 — Seed: отсутствует поле `offsetInLongerWall` в WallAdjacency

**Файл**: `seed/test-apartment.json`, `technical-requirements.md`  
**Серьёзность**: БЛОКЕР (Plan Assembler)

Смежность Коридор D–E (длина 725 мм) и Кухня D–A (длина 3390 мм) — частичное перекрытие. Prisma-схема включает `offsetInLongerWall: Int`, но в seed-данных поле отсутствует. Без него невозможно правильно позиционировать комнаты друг относительно друга.

**Решение**: добавить `offsetInLongerWall` для всех adjacency с частичным перекрытием.

---

### КР-04 — Схема Prisma: отсутствует связь `Wall → WallAdjacency`

**Файл**: `technical-requirements.md` (Prisma schema section)  
**Серьёзность**: БЛОКЕР (ORM query)

В описании модели `Wall` нет обратной связи `adjacencies WallAdjacency[]`. Без неё Prisma не позволит делать `include: { adjacencies: true }` — код в `AdjacencyModule` упадёт в runtime.

**Решение**: добавить в модель `Wall`:
```prisma
adjacenciesAsWallA WallAdjacency[] @relation("WallA")
adjacenciesAsWallB WallAdjacency[] @relation("WallB")
```

---

### КР-05 — API: отсутствует эндпоинт для WallAdjacency

**Файл**: `technical-requirements.md` (API Endpoints section), `sprints/sprint-3.md`  
**Серьёзность**: БЛОКЕР (Sprint 3)

Эндпоинты для `WallAdjacency` не описаны ни в техтребованиях, ни в спринтах, хотя `AdjacencyModule` создаётся в Sprint 3. Frontend не сможет управлять смежностями.

**Решение**: добавить эндпоинты:
- `POST /projects/:id/adjacencies`
- `GET /projects/:id/adjacencies`
- `PATCH /projects/:id/adjacencies/:adjId`
- `DELETE /projects/:id/adjacencies/:adjId`

---

### КР-06 — use-cases.md: UC-03 не описывает шаг ввода углов (Angles)

**Файл**: `docs/use-cases.md`  
**Серьёзность**: БЛОКЕР (UX / Frontend Sprint 2)

UC-03 (Мастер обмера) содержит 7 подшагов 3.1–3.7, но ввод объектов `Angle` не описан как отдельный шаг и не включён в пользовательский сценарий. Разработчик фронтенда не знает, где и как пользователь вводит углы.

**Решение**: добавить подшаг 3.8 «Уточнение углов» или интегрировать ввод угла в шаг 3.2 (добавление стены) с явным описанием UX.

---

### КР-07 — Sprint 2: перегрузка (10 задач, высокий риск)

**Файл**: `sprints/sprint-2.md`

## Архитектурные наблюдения

**З-01 — SVG vs Canvas для плана**  
`PlanAssemblerService` генерирует SVG на сервере. Frontend отображает интерактивный план через Konva.js (Canvas). Нужно определить: SVG используется только для PDF, а Canvas — для интерактива, или SVG конвертируется в Canvas-объекты? Это должно быть задокументировано.

**З-02 — Autosave и конфликты**  
Debounced PUT queue (Zustand persist) не решает конфликты при одновременном редактировании с двух устройств. Для MVP это приемлемо, но нужно явно зафиксировать ограничение в business-requirements.

**З-03 — EXIF cleanup в sharp**  
Удаление EXIF через sharp — правильный подход для GDPR, но нужно убедиться, что orientation EXIF тег обрабатывается до удаления (иначе фото будут повёрнуты неправильно). Это типичная ошибка.

**З-04 — SQLite → PostgreSQL миграция**  
Prisma поддерживает смену провайдера, но SQLite не поддерживает все типы и индексы PostgreSQL. При переходе потребуется ревью схемы. Рекомендуется использовать PostgreSQL с самого начала даже для MVP.

**З-05 — @react-pdf/renderer в NestJS**  
Библиотека создана для React DOM/React Native, не для Node.js SSR без DOM. В NestJS потребуется либо `jsdom` полифилл, либо запуск в отдельном worker_thread. Это нетривиально — нужен spike в Sprint 4.

Основной технический риск — `PlanAssemblerService` (Sprint 3) и `@react-pdf/renderer` в Node.js (Sprint 4). Оба требуют spike-задач до полноценной реализации.
