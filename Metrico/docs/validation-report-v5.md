### CRIT-02. Отсутствие RefreshToken в модели данных

**Расположение:** `technical-requirements.md` (раздел 4, модель User), `sprints/sprint-1.md` (S1-04), `sprints/sprint-3.md` (S3-07)

**Проблема:** В S1-04 реализуется `POST /api/auth/refresh` — обновление access token по httpOnly refresh cookie. В S3-07 явно требуется: «После сброса старый refresh-токен инвалидируется». 

Модель данных содержит `PasswordResetToken`, но **отсутствует таблица `RefreshToken`**. Без хранения refresh token в БД невозможно:
- инвалидировать сессию при смене пароля (S3-07)
- реализовать logout с аннулированием сессии
- защититься от повторного использования украденного refresh token

**Вариант A (stateless JWT):** refresh token не хранится — нельзя инвалидировать.  
**Вариант B (stateful):** нужна таблица `RefreshToken` в Prisma-схеме.

**Исправление:** Добавить в Prisma-схему:
```
RefreshToken
- id (String, @id @default(uuid()))
- userId → User (onDelete: Cascade)
- tokenHash (String, @unique) — bcrypt/SHA-256 хэш токена
- expiresAt (DateTime)
- revokedAt (DateTime?) — null если активен
- createdAt (DateTime)
- @@index([userId])
```
Обновить S1-04 и добавить логику отзыва в S3-07.

### HIGH-01. Формула Гаусса для clockwise-обхода даёт отрицательную площадь

**Расположение:** `technical-requirements.md` (раздел 4, Room, описание RoomsCalcService)

**Текст документа:** «Стены перечислены по часовой стрелке (clockwise). Площадь считается по формуле Гаусса (Shoelace formula).»

**Проблема:** Shoelace formula для clockwise-полигона возвращает **отрицательное** значение. Формула: `A = 0.5 * |Σ(x_i * y_{i+1} - x_{i+1} * y_i)|`. Без взятия модуля (`abs()`) площадь будет отрицательной — это неочевидный баг, который сложно поймать в тестах если тесты также считают «вручную» с тем же знаком.

**Исправление:** Явно добавить в описание RoomsCalcService: «результат формулы берётся по модулю: `computedArea = Math.abs(shoelaceSum) / 2`». Добавить тест: площадь всегда положительная.

### HIGH-05. Отсутствие GET-эндпоинтов для отдельных сущностей второго уровня

**Расположение:** `technical-requirements.md` (раздел 6, API)

**Проблема:** В API нет:
- `GET /api/rooms/:id` — данные конкретной комнаты  
- `GET /api/walls/:id` — данные конкретной стены  
- `GET /api/walls/:wallId/segments` — сегменты стены  
- `GET /api/rooms/:roomId/walls` — стены комнаты  

Вся информация приходит через `GET /api/projects/:id` (fat GET). Это создаёт проблемы:
1. При переходе между подшагами визарда фронтенд перезапрашивает весь проект.
2. При работе с развёрткой стены (S2-07c) нужны данные одной стены — приходится фильтровать из полного ответа.
3. При росте данных (10+ комнат × 4+ стены × 5+ сегментов) fat GET будет медленным.

**Исправление:** Добавить в API раздел минимальный набор read-эндпоинтов или документировать, что fat GET является намеренным решением MVP с указанием ограничений.

### MED-01. Несоответствие cuid() vs uuid() в модели данных

**Расположение:** `technical-requirements.md` (раздел 4)

**Проблема:** Все модели используют `@id @default(uuid())`, кроме `PasswordResetToken`, который использует `@id @default(cuid())`. Это несоответствие без обоснования. UUID и CUID имеют разный формат строки, что может вызвать путаницу в тестах.

**Исправление:** Стандартизировать на `uuid()` везде.

### MED-04. Автосохранение — SQLite нужен WAL-режим

**Расположение:** `technical-requirements.md` (раздел 9), `sprints/sprint-2.md` (S2-08)

**Проблема:** Автосохранение отправляет множество PUT-запросов через очередь с дебаунсом 3 сек. SQLite в default-режиме (journal) блокирует таблицу при записи. При одновременных PUT-запросах (например, сохранение стены + кривизна + сегмент) возможны SQLITE_BUSY ошибки.

**Исправление:** В S1-03 (Prisma-схема) или S1-02 (Docker) добавить включение WAL-режима:
```
datasource db {
  provider = "sqlite"
  url      = env("DATABASE_URL")
}
// В PrismaService.onModuleInit():
await this.$executeRaw`PRAGMA journal_mode=WAL`
```

**Расположение:** `technical-requirements.md` (раздел 11), `sprints/sprint-1.md` (S1-05)

**Исправление:** Уточнить: «CORS разрешён только в dev-режиме (NODE_ENV=development), в production — отключён через Nginx same-origin».
