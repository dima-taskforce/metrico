## 3. Новые критические проблемы

### НК-1: `PasswordResetToken` — схема не определена

**Проблема:** S2-10 (Sprint 2) реализует сброс пароля и требует хранения токенов, но модель `PasswordResetToken` **отсутствует в Prisma-схеме** (TR) и **не включена в миграцию S1-03**.

**Последствие:** Агент, выполняющий S2-10, не имеет эталонной схемы. Он либо придумает собственную структуру, несовместимую с остальными, либо создаст миграцию с неизвестными полями.

**Рекомендация:** Добавить в TR Prisma-схему:

```prisma
model PasswordResetToken {
  id        String    @id @default(cuid())
  userId    String
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  token     String    @unique  // bcrypt-hashed SHA-256 токен
  expiresAt DateTime
  usedAt    DateTime?
  createdAt DateTime  @default(now())

  @@index([userId])
}
```

Добавить `passwordResetTokens PasswordResetToken[]` в модель `User`. Добавить создание этой таблицы в задачу S1-03 (или создать отдельную S2-00 "migration for password reset").

---

### НК-2: `FileStorageService` используется в S1-05, но создаётся в S2-05

**Проблема:** S1-05 (Sprint 1) реализует загрузку чертежа (`blueprintPath`) и описывает использование `FileStorageService`, однако этот сервис создаётся только в S2-05 (Sprint 2). В зависимостях S1-05 указаны только `S1-04`.

**Последствие:** Sprint 1 не компилируется без реализации `FileStorageService`. Агент будет вынужден либо заблокироваться, либо написать inline-логику для файлов, которую потом нужно рефакторить.

**Рекомендация:** Выбрать одно из двух решений:

- **Вариант A (предпочтительный):** Вынести базовый `FileStorageService` (только `saveFile` и `deleteFile`) в **S1-05** как часть задачи. S2-05 расширяет его thumbnail-генерацией и EXIF-очисткой. Обновить зависимость S2-05 на включение S1-05.

- **Вариант B:** Переместить blueprint upload в **S2-05**, в S1-05 сохранять `blueprintPath = null`, добавить отдельный `PATCH /api/projects/:id/blueprint` эндпоинт в S2-05.

## 4. Новые средние проблемы

### НС-1: Поле `Angle.photoPath` есть в модели, но нет эндпоинта загрузки

**Проблема:** TR содержит `Angle.photoPath (String?)` в data model. Однако ни в S2-04 (AnglesModule), ни в S2-05 (PhotosModule) нет эндпоинта `POST /api/angles/:id/photo`.

**Последствие:** Поле либо никогда не будет заполнено (мёртвый код), либо агент-реализатор добавит эндпоинт ad-hoc без валидации, вне FileStorageService pipeline.

**Рекомендация:** Добавить в S2-05 эндпоинт `POST /api/angles/:id/photo` по аналогии с `/windows/:id/photo`. Либо явно задокументировать в TR, что `photoPath` — зарезервировано для post-MVP и `null` по умолчанию.

---

### НС-2: S4-05 зависит от S3-03, но в зависимостях указан только S4-02

**Проблема:** S4-05 добавляет статус `COMPLETED` при успешном вызове `plan/assemble`. Эндпоинт `POST /api/projects/:id/plan/assemble` реализуется в **S3-03** (PlanAssemblerService). S4-05 в зависимостях указывает только `S4-02`.

**Последствие:** Агент S4-05 может не знать о существующей архитектуре PlanModule, дублировать логику или создать конфликт модулей.

**Рекомендация:** В S4-05 добавить в зависимости `S3-03`. Явно указать: "модифицировать `PlanAssemblerService.assemble()` — добавить вызов `projectsService.updateStatus(id, 'COMPLETED')` после успешной сборки".

---

### НС-3: S5-01 (OAuth) не описывает обработку конфликтов провайдеров

**Проблема:** TR Section 6 описывает конфликт: пользователь с `LOCAL` учётной записью пытается войти через OAuth с тем же email — должна вернуться специфическая ошибка. S5-01 этот сценарий не упоминает.

**Последствие:** Агент реализует happy-path OAuth без учёта конфликта, что приведёт к дублированию аккаунтов или неинформативным ошибкам.

**Рекомендация:** Добавить в S5-01 явный acceptance-критерий: "при OAuth входе с email, уже зарегистрированным через LOCAL, возвращать HTTP 409 с `{ code: 'EMAIL_PROVIDER_CONFLICT', provider: 'local' }`". Реализовать в `AuthService.validateOAuthUser()`.

## 5. Новые минорные проблемы

### НМ-1: Нумерация задач в Sprint 1 — S1-11 стоит перед S1-10

**Проблема:** В файле `sprint-1.md` задача S1-11 (Wizard step 1 UI) расположена **перед** S1-10 (GitHub Actions CI). Нарушает ожидаемый порядок чтения.

**Рекомендация:** Переставить задачи в правильный порядок (S1-10 → S1-11) или перенумеровать.

---

### НМ-2: `FloorPlanLayout.layoutJson` хранит координаты в пикселях

**Проблема:** PlanAssemblerService генерирует `x/y` координаты для комнат в пикселях (зависят от разрешения экрана). При отображении на разных устройствах план будет рендериться по-разному.

**Рекомендация:** Хранить координаты в **миллиметрах** (реальная система координат из Wall.length). Масштабирование пиксель/мм вычислять на клиенте при рендеринге Konva. Это также позволит корректно рассчитывать PDF-масштаб в S4-02.

---

### НМ-3: Офлайн-автосохранение — потеря данных при закрытии браузера

**Проблема:** S2-08 буферизует изменения в Zustand store. Если браузер закрывается в офлайн-режиме с непустым буфером, все несинхронизированные изменения теряются. Это архитектурное ограничение нигде не задокументировано.

**Рекомендация:** Добавить в S2-08 использование `localStorage` как persistence layer для буфера (`zustand/middleware/persist`). При инициализации приложения — проверять localStorage на наличие несинхронизированных операций и предлагать синхронизацию. Либо явно задокументировать как known limitation в BR/TR.

## 6. Открытая проблема: objectType enum (М-3)

**Проблема:** `objectType` enum в TR содержит `APARTMENT | STUDIO | HOUSE` (3 значения). Business requirements и UI в S1-11 добавляют **апартаменты** (`APARTMENTS`) как отдельный тип.

**Рекомендация:** Добавить в TR Prisma-схему:

```prisma
enum ObjectType {
  APARTMENT
  STUDIO
  APARTMENTS  // апартаменты (не квартира)
  HOUSE
}
```

Обновить описание UC-02 в use-cases.md.
