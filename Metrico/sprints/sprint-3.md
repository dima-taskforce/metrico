## S3-01. CRUD связей стен (AdjacencyModule)

**Агент:** nestjs-expert

**Описание:**
Реализовать:
- `POST /api/projects/:projectId/adjacencies` — создать связь (wallAId, wallBId, hasDoorBetween, doorOpeningId?).
- `GET /api/projects/:projectId/adjacencies` — список всех связей проекта.
- `DELETE /api/adjacencies/:id` — удалить связь.

Стены идентифицируются парами углов (cornerFrom-cornerTo). Например, стена "A-B" в комнате X примыкает к стене "C-D" в комнате Y.

Валидация (3 инварианта из technical-requirements.md):
1. wallA и wallB принадлежат разным комнатам одного проекта.
2. Обратная связь (wallB→wallA) не создаётся как отдельная запись — проверка в обе стороны.
3. Если hasDoorBetween = true, doorOpeningId должен быть не null.

Unique constraint: @@unique([wallAId, wallBId]).

**Тесты:**
- Создание связи. Дубликат → ошибка. Стены из одной комнаты → ошибка. Обратная связь → ошибка. hasDoorBetween=true без doorOpeningId → ошибка. Удаление.

**Зависимости:** Sprint 2 завершён.

## S3-02. Сервис расчётов (RoomsCalcService)

**Агент:** nestjs-expert

**Описание:**
Реализовать `RoomsCalcService`:
- `computeArea(room)` — площадь помещения по формуле Гаусса (из длин стен и углов). Для прямоугольников — упрощённая формула.
- `computePerimeter(room)` — сумма Wall.length по всем стенам комнаты (перекрёстная проверка: должна совпадать с суммой WallSegment.length по всем сегментам).
- `computeVolume(room)` — площадь × средняя высота потолка ((h1 + h2) / 2).
- `computeCurvatureAvg(wall)` — среднее кривизны (3 точки).
- `computeCurvatureDeviation(wall)` — максимальное отклонение от среднего.

Стены идентифицируются парами углов (cornerFrom/cornerTo).

`GET /api/projects/:id` возвращает вычисляемые поля в каждой комнате.

**Тесты:**
- Прямоугольная комната: площадь = длина × ширина.
- Г-образная комната: корректная площадь.
- Объём: проверка с двумя разными высотами потолка.
- Кривизна: среднее и отклонение.
- Сравнение с эталонными значениями из `seed/test-apartment.json`.

**Зависимости:** S3-01, S2-06 (seed-данные).

## S3-03. Алгоритм сборки плана (PlanAssemblerService)

**Агент:** nestjs-expert

**Описание:**
Реализовать `PlanAssemblerService`:
1. Построить полигон каждой комнаты из длин стен и углов.
2. Разместить первую комнату в начале координат.
3. Для каждой WallAdjacency — позиционировать смежную комнату так, чтобы связанные стены совпали. Стены идентифицируются парами углов (cornerFrom/cornerTo), например стена "A-B" примыкает к стене "C-D".
4. Несвязанные комнаты — разместить рядом с пометкой.
5. Сохранить `layoutJson` (позиции/повороты) и `svgData`.

API:
- `POST /api/projects/:id/plan/assemble` — запустить сборку. Ответ: `{ layoutJson, svgData }`.
- `GET /api/projects/:id/plan` — получить текущий план.
- `PUT /api/projects/:id/plan` — сохранить ручные правки layoutJson.

**Тесты:**
- Прямоугольная квартира (2 комнаты, 1 связь): корректные координаты.
- Seed-квартира: все комнаты размещены, SVG валиден.
- Несвязанная комната: помечена.

**Зависимости:** S3-01, S3-02.

## S3-07. Сброс пароля (PasswordResetModule)

**Агент:** nestjs-expert

**Описание:**
Перенесено из Sprint 2 (бывший S2-10) для разгрузки спринта.

Реализовать:
- `POST /api/auth/forgot-password` — генерация токена сброса, отправка email со ссылкой.
- `POST /api/auth/reset-password` — приём токена + нового пароля, валидация TTL (1 час), одноразовое использование.
- PasswordResetToken: хранение bcrypt-хэша от SHA-256 токена, поле `usedAt` для предотвращения повторного использования.
- SMTP-транспорт через `@nestjs-modules/mailer` + Handlebars шаблон письма.

**Критерии приёмки:**
- Пользователь получает email со ссылкой на сброс пароля.
- Ссылка работает однократно и истекает через 1 час.
- После сброса **все** активные RefreshToken пользователя инвалидируются: `UPDATE RefreshToken SET revokedAt = now() WHERE userId = ? AND revokedAt IS NULL`. Реализуется в AuthService.revokeAllRefreshTokens(userId).
- При несуществующем email — HTTP 200 (не раскрываем наличие аккаунта).

**Тесты:**
- Полный цикл: forgot → email → reset → login с новым паролем.
- После сброса старый refresh cookie возвращает 401 (токен revoked).
- Повторное использование токена → 400.
- Истёкший токен → 400.

**Зависимости:** S1-04 (AuthModule с RefreshToken).

## S3-09. Автосохранение

**Агент:** fullstack

**Описание:**
Перенесено из Sprint 2 (бывший S2-08) для разгрузки спринта.

Реализовать:
- На фронте: хук `useAutoSave` — дебаунс 3 сек после ввода. Индикатор: «Сохранено» / «Сохранение...» / «Нет связи».
- **Стратегия сохранения:** autosave вызывает существующие CRUD endpoints (PUT /api/walls/:id, PUT /api/windows/:id, PUT /api/doors/:id, PUT /api/elements/:id, PUT /api/angles/:id, PUT /api/segments/:id, PUT /api/rooms/:id) через очередь запросов. Каждый изменённый ресурс сохраняется отдельным запросом. Это исключает конфликт семантики с REST CRUD и race conditions.
- Очередь: изменения группируются по ресурсу (wall, window, etc.), отправляются последовательно. При ошибке одного запроса — повтор через 5 сек, остальные не блокируются.
- При потере соединения: копить изменения в Zustand, отправить при восстановлении (в порядке очереди).
- **Persistence:** использовать `zustand/middleware/persist` с `localStorage` для буфера несинхронизированных операций. При инициализации приложения проверять localStorage — если есть несинхронизированные операции, показывать баннер «Есть несохранённые изменения» и предлагать синхронизацию. Это защищает от потери данных при закрытии браузера в офлайн-режиме.

**Тесты:**
- Дебаунс срабатывает корректно. Индикатор меняет состояние. Офлайн-режим: данные не теряются. Очередь отправляет запросы последовательно.

**Зависимости:** S2-07, S2-04b (завершены в Sprint 2).
