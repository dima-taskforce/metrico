## S2-05. Загрузка фото (PhotosModule + FileStorageService)

**Агент:** nestjs-expert

**Описание:**
Реализовать:
- `POST /api/rooms/:roomId/photos` — загрузка фото комнаты (multipart). Query: `?type=OVERVIEW_BEFORE|OVERVIEW_AFTER|DETAIL`.
- `POST /api/elements/:elementId/photo` — фото элемента.
- `POST /api/windows/:windowId/photo` — фото окна.
- `POST /api/doors/:doorId/photo` — фото двери.
- `POST /api/angles/:angleId/photo` — фото угла.
- `DELETE /api/photos/:id` — удаление (из БД + с диска).
- `FileValidationPipe`: MIME (image/jpeg, image/png, image/webp), размер ≤ 10 МБ.
- **Расширение `FileStorageService`** (базовая версия создана в S1-05): добавить EXIF-очистку через `sharp`, генерацию thumbnail (400px по большей стороне), сохранение в `uploads/{userId}/{projectId}/{roomId}/`.
- Статика фото через Nginx (настройка location в nginx.conf).

**Тесты:**
- Загрузка валидного фото. Отклонение невалидного MIME. Отклонение > 10 МБ. Генерация thumbnail. Очистка EXIF. Удаление файла с диска.

**Зависимости:** S1-05, S2-01.

## S2-08. Автосохранение

**Агент:** fullstack

**Описание:**
Реализовать:
- На фронте: хук `useAutoSave` — дебаунс 3 сек после ввода. Индикатор: «Сохранено» / «Сохранение...» / «Нет связи».
- **Стратегия сохранения:** autosave вызывает существующие CRUD endpoints (PUT /api/walls/:id, PUT /api/windows/:id, PUT /api/doors/:id, PUT /api/elements/:id, PUT /api/angles/:id, PUT /api/rooms/:id) через очередь запросов. Каждый изменённый ресурс сохраняется отдельным запросом. Это исключает конфликт семантики с REST CRUD и race conditions.
- Очередь: изменения группируются по ресурсу (wall, window, etc.), отправляются последовательно. При ошибке одного запроса — повтор через 5 сек, остальные не блокируются.
- При потере соединения: копить изменения в Zustand, отправить при восстановлении (в порядке очереди).
- **Persistence:** использовать `zustand/middleware/persist` с `localStorage` для буфера несинхронизированных операций. При инициализации приложения проверять localStorage — если есть несинхронизированные операции, показывать баннер «Есть несохранённые изменения» и предлагать синхронизацию. Это защищает от потери данных при закрытии браузера в офлайн-режиме.

**Тесты:**
- Дебаунс срабатывает корректно. Индикатор меняет состояние. Офлайн-режим: данные не теряются. Очередь отправляет запросы последовательно.

**Зависимости:** S2-07.

## S2-10. Сброс пароля (email)

**Агент:** fullstack

**Описание:**
Реализовать:
- Настройка SMTP-транспорта (nodemailer + NestJS MailerModule). Провайдер из .env (SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS).
- `POST /api/auth/forgot-password` — генерация одноразового токена (UUID, TTL 1 час), сохранение в БД (новая модель `PasswordResetToken`), отправка email со ссылкой.
- `POST /api/auth/reset-password` — проверка токена, установка нового пароля.
- Фронтенд: страница «Забыл пароль» + страница «Новый пароль» (по ссылке из email).

**Тесты:**
- Генерация токена. Истёкший токен → ошибка. Успешный сброс. Email отправлен (mock).

**Зависимости:** S1-04.
